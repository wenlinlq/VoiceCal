# 一、项目概述

本项目是一个以语音交互为核心的日历管理工具。用户通过说话即可完成日程添加、查看、删除、修改，前端实时展示
ASR 识别文字，后端完成意图识别、日历操作和 TTS 语音反馈。

| **能力**  | **用户示例**               | **后端处理**                                                     |
|-----------|----------------------------|------------------------------------------------------------------|
| 语音添加  | 明天下午三点开产品评审会   | ASR -\> create_event 意图 -\> 时间解析 -\> 创建日程 -\> TTS 播报 |
| 语音查看  | 今天有什么安排             | ASR -\> query_events 意图 -\> 查询列表 -\> 文本/语音总结         |
| 语音删除  | 删除今天下午三点的会       | ASR -\> delete_event 意图 -\> 匹配事件 -\> need_confirm -\> 删除 |
| 语音修改  | 把明天的会改到后天上午十点 | ASR -\> update_event 意图 -\> 匹配事件 -\> need_confirm -\> 修改 |
| 手动 CRUD | 前端表单增删改查           | HTTP REST 直接调用 Calendar Service                              |

# 三、ASR 和 TTS 是否可以用 Qwen？

结论：可以用，但要区分“Qwen 文本大模型接口”和“Qwen
语音实时接口”。ASR/TTS 不建议直接走普通文本 Chat
接口，而应接入阿里云百炼 / DashScope Model Studio
的实时语音识别和实时语音合成能力。

| **能力**   | **Qwen 可行性**                                                                          | **推荐策略**                                                                            |
|------------|------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------|
| 流式 ASR   | 可用。Qwen-ASR Realtime 支持 WebSocket 音频流输入和低延迟文本输出，适合语音助手场景。    | MVP 优先 Qwen-ASR Realtime；若 SDK/区域/成本受限，用 FunASR/Paraformer 本地或云端兜底。 |
| 流式 TTS   | 可用。Qwen-TTS Realtime 支持 WebSocket 流式输入和流式输出；也可使用 CosyVoice 实时合成。 | MVP 可先返回 base64 MP3；体验优化阶段改为 tts_chunk 流式音频。                          |
| Agent 推理 | 可用。Qwen/DeepSeek 都可以通过 OpenAI-compatible 或 DashScope SDK 调用。                 | 若你已经决定 DeepSeek V4 做 NLU，可继续保留；Qwen 主要承担 ASR/TTS。                    |

落地建议：ASR/TTS 用 Qwen 语音实时接口，Agent 推理可用 DeepSeek
V4。这样既满足流式语音要求，也保留 DeepSeek 在中文意图推理上的使用。

# 七、WebSocket 流式语音接口

## 7.1 连接地址

GET /ws/voice?user_id=10001&session_id=uuid&timezone=Asia/Shanghai

| **参数**   | **类型** | **必填** | **说明**                     |
|------------|----------|----------|------------------------------|
| user_id    | string   | 否       | 用户 ID，MVP 可匿名          |
| session_id | string   | 否       | 前端生成 UUID，后端也可补齐  |
| timezone   | string   | 否       | 用户时区，默认 Asia/Shanghai |

## 7.2 消息帧格式

\[4 字节 header\] + \[payload\]  
  
header = 0x01: payload 为 PCM 二进制音频  
header = 0x02: payload 为 UTF-8 JSON 控制消息

| **Header** | **类型** | **Payload** | **说明**                                       |
|------------|----------|-------------|------------------------------------------------|
| 0x01       | audio    | PCM bytes   | 前端每 500ms-1000ms 发送一片音频               |
| 0x02       | json     | UTF-8 JSON  | start/stop/confirm/cancel/heartbeat 等控制消息 |

## 7.3 前端发送控制消息

| **action** | **说明**       | **后端行为**                                       |
|------------|----------------|----------------------------------------------------|
| start      | 开始录音       | 初始化 ASR 流、Agent 上下文，返回 status/listening |
| stop       | 停止录音       | 结束 ASR 流，返回 final 文本，进入 Agent 处理      |
| confirm    | 确认高风险操作 | 执行 pending action，例如删除/修改                 |
| cancel     | 取消高风险操作 | 清除 pending action，返回 OPERATION_CANCELLED      |
| heartbeat  | 心跳包         | 返回 heartbeat + server_time                       |

// start  
{  
"type": "control",  
"action": "start",  
"request_id": "req_001",  
"timestamp": 1780000000000  
}  
  
// stop  
{  
"type": "control",  
"action": "stop",  
"request_id": "req_001",  
"timestamp": 1780000001000  
}  
  
// confirm  
{  
"type": "control",  
"action": "confirm",  
"request_id": "req_002",  
"confirm_id": "confirm_001"  
}

## 7.4 后端推送消息类型

| **type**     | **用途**          | **关键字段**                                           |
|--------------|-------------------|--------------------------------------------------------|
| asr          | ASR 实时识别结果  | is_final, text, confidence                             |
| status       | 语音/Agent 状态   | state=listening/thinking/speaking/waiting_confirm/idle |
| need_confirm | 需要用户确认      | confirm_id, action, message, event/events              |
| tts          | 一次性 TTS 音频   | format, audio, text                                    |
| tts_chunk    | 流式 TTS 音频分片 | seq, is_final, audio, format, sample_rate              |
| result       | 业务操作结果      | success, code, message, data                           |
| error        | 错误信息          | code, message, detail                                  |
| heartbeat    | 心跳响应          | timestamp, server_time                                 |

## 7.5 ASR 流式消息

// partial text  
{  
"type": "asr",  
"request_id": "req_001",  
"is_final": false,  
"text": "明天下午三点",  
"confidence": 0.92  
}  
  
// final text  
{  
"type": "asr",  
"request_id": "req_001",  
"is_final": true,  
"text": "明天下午三点开产品评审会",  
"confidence": 0.96  
}

ASR partial 用于前端实时字幕；final 才进入 Agent 业务处理。不要用
partial 直接创建/删除日程。

## 7.6 TTS 流式消息设计

为了满足“语音播报也要流式”的要求，后端建议支持两种返回方式：MVP 使用 tts
一次性 base64 MP3；正式流式体验使用 tts_chunk
多片返回。前端已有播放队列设计，可以直接按 seq 顺序播放。

// MVP：一次性返回，简单稳定  
{  
"type": "tts",  
"request_id": "req_001",  
"format": "mp3",  
"audio": "base64_audio_string",  
"text": "已为你添加明天下午三点的产品评审会"  
}  
  
// 优化版：流式返回，可边生成边播放  
{  
"type": "tts_chunk",  
"request_id": "req_001",  
"seq": 1,  
"is_final": false,  
"format": "mp3",  
"sample_rate": 24000,  
"audio": "base64_audio_chunk",  
"text_delta": "已为你添加"  
}  
  
{  
"type": "tts_chunk",  
"request_id": "req_001",  
"seq": 2,  
"is_final": true,  
"format": "mp3",  
"sample_rate": 24000,  
"audio": "base64_audio_chunk",  
"text_delta": "明天下午三点的产品评审会"  
}

| **字段**    | **说明**                                         |
|-------------|--------------------------------------------------|
| seq         | 分片序号，从 1 开始，前端用于排序                |
| is_final    | 是否最后一片，true 后前端可结束播放状态          |
| format      | mp3/pcm/opus，MVP 推荐 mp3                       |
| sample_rate | 输出采样率，通常 24000 或 48000，视 TTS 服务而定 |
| audio       | base64 编码后的音频分片                          |
| text_delta  | 本片对应的文字，可选，用于字幕同步               |

# 十一、流式 ASR 处理方案

| **阶段**    | **后端动作**                                | **输出给前端**                           |
|-------------|---------------------------------------------|------------------------------------------|
| start       | 创建 ASR realtime session，绑定 request_id  | status/listening                         |
| audio chunk | 校验 PCM；转发给 Qwen-ASR；监听增量识别结果 | asr is_final=false                       |
| stop        | 发送结束信号；等待最终文本                  | asr is_final=true + status/thinking      |
| fail        | 捕获 ASR 错误和超时                         | error ASR_FAILED 或 INVALID_AUDIO_FORMAT |

class ASRService:  
async def start_stream(self, session_id: str, sample_rate=16000): ...  
async def send_audio(self, session_id: str, pcm: bytes): ...  
async def receive_partial(self, session_id: str) -\>
AsyncIterator\[ASRResult\]: ...  
async def finish_stream(self, session_id: str) -\> ASRResult: ...  
async def close_stream(self, session_id: str): ...

| **校验项** | **规则**                                            |
|------------|-----------------------------------------------------|
| 采样率     | 16000 Hz                                            |
| 声道       | mono                                                |
| 位深       | 16 bit                                              |
| 分片大小   | 500ms-1000ms，空分片丢弃                            |
| 异常处理   | 未 start 收到音频则丢弃或返回 INVALID_SESSION_STATE |

# 十二、流式 TTS 处理方案

| **模式**           | **适合阶段** | **优点**               | **缺点**                         |
|--------------------|--------------|------------------------|----------------------------------|
| 一次性 base64 MP3  | MVP 演示     | 前端兼容好，实现简单   | 首包延迟较高，不是真正边生成边播 |
| tts_chunk 流式 MP3 | 正式联调     | 前端可边接收边排队播放 | 前端要处理队列、乱序、结束标记   |
| PCM/Opus 流式      | 低延迟优化   | 延迟低，适合实时对话   | 小程序播放处理更复杂             |

@app.websocket('/ws/voice')  
async def voice_ws(websocket: WebSocket):  
await websocket.accept()  
session = await session_service.create(websocket.query_params)  
  
try:  
while True:  
raw = await websocket.receive_bytes()  
header = raw\[:4\]  
payload = raw\[4:\]  
  
if header == AUDIO_HEADER:  
if not session.is_listening:  
continue  
if not audio_validator.is_valid_pcm(payload):  
await send_error(websocket, 'INVALID_AUDIO_FORMAT')  
continue  
await asr_service.send_audio(session.id, payload)  
async for partial in asr_service.poll_partial(session.id):  
await websocket.send_json({  
'type': 'asr', 'request_id': session.request_id,  
'is_final': False, 'text': partial.text  
})  
  
elif header == JSON_HEADER:  
data = json.loads(payload.decode('utf-8'))  
action = data.get('action')  
  
if action == 'start':  
await asr_service.start_stream(session.id)  
session.state = 'listening'  
await websocket.send_json({'type': 'status', 'state': 'listening'})  
  
elif action == 'stop':  
final = await asr_service.finish_stream(session.id)  
await websocket.send_json({'type': 'asr', 'is_final': True, 'text':
final.text})  
await websocket.send_json({'type': 'status', 'state': 'thinking'})  
agent_result = await agent_service.handle(final.text, session)  
await handle_agent_result(websocket, session, agent_result)  
  
elif action == 'confirm':  
result = await session_service.confirm(session.id,
data\['confirm_id'\])  
await websocket.send_json(result)  
  
elif action == 'cancel':  
await session_service.cancel(session.id, data.get('confirm_id'))  
await websocket.send_json({'type': 'result', 'success': True, 'code':
'OPERATION_CANCELLED'})  
  
elif action == 'heartbeat':  
await websocket.send_json({'type': 'heartbeat', 'server_time':
now_iso()})  
finally:  
await asr_service.close_stream(session.id)  
await session_service.close(session.id)
